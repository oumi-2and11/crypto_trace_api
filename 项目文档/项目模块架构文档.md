# 项目模块架构文档：信息安全密码算法接口化实现与可测验证平台（Python + Flask）

> 目标：在 Python + Flask 下实现课程要求的密码算法（从零实现，不调用第三方密码算法库），并提供统一 API 供第三方调用，返回结果与执行状态码，同时做到输入输出与执行过程“可见、可测”。  
> 目录组织风格：参考仓库 `oumi-2and11/beacon_proc_risk`（根目录入口 + `webapp/` 应用层 + `utils/` 工具/能力层 + `项目文档/` 文档层）。

---

## 1. 总体架构概览

### 1.1 分层思想
- **入口层（Root）**：应用启动、环境配置、依赖声明、容器化运行。
- **应用层（webapp/）**：Flask App 初始化、路由/接口、参数校验、统一响应、（可选）页面展示。
- **能力层（utils/）**：密码算法实现、编码/字节工具、trace 追踪、测试向量、自检逻辑。
- **文档层（项目文档/）**：作业报告素材、实现计划、接口说明、测试记录等。

### 1.2 数据流（API 调用链路）
客户端（第三方程序/curl）  
→ `webapp/views/*`（路由：解析 JSON、校验参数、决定是否 trace）  
→ `utils/*`（算法执行：返回 result + trace + 统计）  
→ `webapp/views/*`（封装统一响应：status_code/result/trace/time_ms）  
→ 客户端

## 3. 核心模块职责说明

### 3.1 Root 层
- `app.py`
  - 作为程序入口：读取配置、初始化 Flask App、启动服务。
- `config.py`
  - 集中管理运行配置：debug、host/port、trace 默认开关、最大 trace 输出大小限制等。
- `requirements.txt`
  - 仅包含 Flask 与工程化依赖（不包含现成密码算法库）。
- `docker-compose.yml`（可选）
  - 用于演示环境快速启动（也方便你截图写报告“运行环境/部署方式”）。

### 3.2 `webapp/`（应用层）
- `webapp/__init__.py`
  - `create_app()`：注册蓝图（Blueprint）、注册错误处理器、统一 JSON 响应格式。
- `webapp/views/*_api.py`
  - 每类算法一组接口文件，便于按评分项写报告和按功能演示。
  - 职责：
    1) 参数解析（encoding、trace、算法名、key/iv、mode 等）
    2) 参数校验（长度、格式、范围）
    3) 调用 `utils/` 对应算法函数
    4) 统一封装返回：`status_code/status_message/result/trace/time_ms`

### 3.3 `utils/`（能力层：算法与基础设施）
- `utils/common/encoding.py`
  - 管理 `hex/base64/text(utf-8)` → bytes 的统一解析；
  - 管理 bytes → hex/base64 的统一输出；
  - 这是“输入输出可见及可测”的基础模块。
- `utils/common/trace.py`
  - `TraceCollector`：提供 `add_step(name, data)`，用于记录关键中间值；
  - 支持 trace 采样、字段截断（防止输出过大）、脱敏展示（例如 key 只显示前后若干字节）。
- `utils/common/validation.py`
  - 统一错误类型与状态码映射（便于接口层直接返回）。
- `utils/symmetric/`, `utils/hash/`, `utils/mac_kdf/`, `utils/asymmetric/`
  - 存放“从零实现”的算法代码；
  - Level 2 trace 的算法（AES/SHA256/RSA/ECDSA）在实现中主动调用 TraceCollector 记录步骤。

### 3.4 `utils/tests/`（可测性证明）
- `vectors.py`
  - 存放测试向量（输入、参数、期望输出 expected）。
- `selftest.py`
  - 返回结构建议：按算法列出 PASS/FAIL、actual/expected 摘要、耗时；
  - 同时为 `/api/selftest` 提供后端能力。
- `cases/test_*.py`
  - pytest 单元测试（可选但强烈推荐；报告中可截图证明可测）。

## 4. Trace（执行过程可见）设计规范

### 4.1 Trace 开关
- 请求参数：`trace=true/false`
- 默认：trace 关闭（只返回最终结果），避免输出过大；
- 报告截图与展示时打开 trace。

### 4.2 Level 1（所有算法具备）
trace 至少包含：
- `algorithm`, `operation`
- `input_summary`：长度、编码、前若干字节 hex
- `params`：模式/迭代次数/输出长度等
- `output_summary`：长度、前若干字节 hex
- `time_ms`

### 4.3 Level 2（AES / SHA256 / RSA / ECDSA）
- AES：轮密钥扩展摘要 + 每轮 state 采样（可配置采样轮数）
- SHA256：padding 信息、W[t] 摘要、a..h 寄存器采样
- RSA：keygen 的 p/q 位数与 n 位数、e/d 生成摘要、签名/验签关键步骤摘要
- ECDSA：曲线参数标识、点运算/模逆关键中间量摘要、(r,s) 与验签判断摘要

> 重点：trace “可解释 + 可复现 + 可截图”，不追求输出全部内部数据（否则太大且不利于接口调用）。

## 5. 状态码（接口可用性要求）
建议统一三段式：
- `0`：成功
- `1xxx`：参数错误（缺字段、格式不对、长度不对、encoding 不支持）
- `2xxx`：算法执行错误（数学不可逆、验签失败、padding 错误等）
- `9xxx`：系统错误（未捕获异常）

接口返回示例字段（所有接口一致）：
- `status_code`
- `status_message`
- `result`
- `trace`（可选）
- `time_ms`
