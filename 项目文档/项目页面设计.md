# 项目页面设计：信息安全密码算法接口化实现与可测验证平台（蓝色系 UI）

> 面向：Python + Flask + Jinja2 模板  
> 目标：让“输入输出可见、执行过程可见、可测（自检向量）”在页面上**直观展示**，并支持四大类算法的快速切换与演示。  
> UI 风格：偏蓝色系（科技感、干净、低饱和背景 + 高对比蓝色主色）。

---

## 1. 信息架构与导航方案选择

你提出的两种导航方式里，我更推荐 **方案 B（两层导航）**，原因：

- 顶部导航只放“四类算法”，结构稳定、不会因为算法数量多而拥挤；
- 进入某一类后，侧栏/顶栏列出该类全部算法，用户在“同一类”里切换成本低；
- 更符合“平台/实验台”的交互模型，也更利于后续扩展（加测试向量、加 trace 视图、加对比视图等）。

### 方案 B：两层导航（推荐）
- 顶部导航（`layout.html`）：  
  **首页** / **对称加密** / **哈希&派生** / **编码** / **公钥&签名** / **自检(SelfTest)** / **关于**
- 点击某一类（例如“哈希&派生”）进入该类默认算法页（例如 SHA256）。
- 该类页面使用 `layout_<category>.html`（继承 `layout.html`），增加：
  - 左侧边栏（推荐）或顶部二级导航（可选）
  - 列出该类所有算法入口

### 方案 A：顶部下拉菜单（不首选）
- 顶部导航每个类下拉出算法列表，直接跳转。
- 缺点：算法多时下拉很长；“类内切换”没有上下文；移动端下拉体验一般。

> 结论：采用“`layout.html` + 类别 layout 二级模板 + 每个算法页面”的继承结构是合理且可维护的，你的想法很对。

## 3. 每个“算法页面”的统一布局（强烈建议统一）

每个算法页面分为 3 个核心区块：**输入区 → 输出区 → Trace/过程区**（Trace 可折叠）。

### 3.1 输入区（Input）
- 输入方式：Text / Hex / Base64（单选）
- 必要参数：key/iv/nonce/iter 等（按算法动态显示）
- 操作按钮：执行 / 清空 / 填充示例用例

### 3.2 输出区（Result）
- 结果以 Tabs 展示：
  - `hex`
  - `base64`
  - `text`（若可解码）
- “复制”按钮
- 耗时 `time_ms`

### 3.3 Trace 区（Execution Trace）
- 开关：`trace=true`（页面按钮或 checkbox）
- 展示方式：步骤列表（timeline 风格）或折叠面板（accordion）
- 对 Level 2 算法（AES/SHA256/RSA/ECDSA）：
  - 额外提供“采样粒度”选择（如展示前 N 轮/关键轮）
  - 中间值展示建议默认截断（例如只展示前 32/64 hex 字符）

## 4. 蓝色系 UI 规范（与参考项目区分开风格）

### 4.1 颜色（建议采用 CSS 变量）
- 主色（Primary）：`#2563EB`（蓝）
- 深主色（Primary Dark）：`#1D4ED8`
- 强调色（Accent / Cyan）：`#06B6D4`
- 背景（Background）：`#0B1220`（深蓝黑，偏科技）
- 面板底色（Panel）：`#111A2E`
- 边框线（Border）：`rgba(255,255,255,0.08)`
- 文本主色：`#E5E7EB`
- 文本次色：`#9CA3AF`
- 成功/警告/失败：
  - Success：`#22C55E`
  - Warning：`#F59E0B`
  - Danger：`#EF4444`

> 如果你更想“浅色蓝白风”，也可把 Background 改成 `#F6F8FF`，Panel 改成白色；但深色科技风更像“安全/密码实验台”。

### 4.2 字体与排版
- 字体栈：`system-ui, -apple-system, Segoe UI, Microsoft YaHei, sans-serif`
- 卡片圆角：14px
- 输入/按钮圆角：10px
- 布局宽度：主体容器 max-width 1100px（大屏不会太散）

### 4.3 组件规范
- 顶部导航：sticky，背景半透明 + blur（但颜色走蓝黑，不走薄荷绿）
- Button：主按钮蓝色渐变（可选），hover 提升亮度
- Card：轻微阴影 + 细边框
- Code block：深色底 + 等宽字体，适合展示 hex/trace

## 5. 模板继承与 `templates/` 目录结构设计（按你说的多级 layout）

建议 `templates/` 目录这样组织（清晰、易维护、和算法分组强绑定）：

```text
webapp/templates/
├── layout.html                      # 全局：顶栏导航、全局 CSS、footer、flash/toast
├── partials/
│   ├── _navbar.html                 # 顶栏导航（可拆）
│   ├── _footer.html
│   ├── _form_helpers.html           # 表单组件（输入框、选择器等宏/片段）
│   └── _trace_panel.html            # trace 展示面板（通用）
│
├── index.html
├── about.html
├── selftest.html
│
├── layouts/
│   ├── layout_symmetric.html        # 继承 layout.html + 左侧栏（AES/SM4/RC6）
│   ├── layout_hash.html             # 继承 layout.html + 左侧栏（SHA* / HMAC / PBKDF2）
│   ├── layout_encoding.html         # 继承 layout.html + 左侧栏（Base64/UTF-8）
│   └── layout_asymmetric.html       # 继承 layout.html + 左侧栏（RSA/ECC/ECDSA）
│
├── symmetric/
│   ├── aes.html
│   ├── sm4.html
│   └── rc6.html
│
├── hash/
│   ├── sha1.html
│   ├── sha256.html
│   ├── sha3.html
│   ├── ripemd160.html
│   ├── hmac_sha1.html
│   ├── hmac_sha256.html
│   └── pbkdf2.html
│
├── encoding/
│   ├── base64.html
│   └── utf8.html
│
└── asymmetric/
    ├── rsa.html
    ├── rsa_sha1.html
    ├── ecc.html
    └── ecdsa.html
```

### 关键点
- `layout.html`：只负责“一级导航 + 全局样式 + 页面容器”，不关心算法细节。
- `layouts/layout_<category>.html`：只负责“二级导航（侧栏/顶栏）+ 内容区域占位”。
- 每个算法页面只写该算法的表单字段与结果展示，复用 partials 的 trace 面板。

## 7. 与后端接口的对应关系（页面如何调用 API）

页面建议用两种模式之一：
1) **纯后端渲染**：表单提交到后端 view，后端调用算法并渲染结果（实现简单）
2) **AJAX 调用 API**（推荐一点点）：点击“执行”后 fetch `/api/...`，把 JSON 结果填入页面（体验更像平台）

为了满足作业“第三方接口调用”，即使你做了页面，也应保留独立的 `/api/*` 路由。
