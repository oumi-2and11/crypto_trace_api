## 3. 总体技术路线（Python + Flask）
### 3.1 系统形态
- 采用 **Flask** 提供 RESTful API 服务。
- 所有算法以 Python 编写“从零实现”（不使用 `cryptography` / `pycryptodome` / OpenSSL binding 等现成算法实现来直接出结果）。
- 对外统一 JSON 输入输出，便于第三方程序调用（Python/Java/Node/curl 均可）。

### 3.2 模块化实现思路（建议在报告“系统设计”中画架构图）
- `api/`：Flask 路由层（参数解析、状态码、返回结构）
- `algorithms/`：算法实现层（AES/SM4/RC6、各类哈希、公钥算法）
- `tracing/`：执行过程追踪层（trace 数据结构、采样策略、脱敏展示）
- `tests/`：测试向量、自检与回归测试（可测性证明）
- `utils/`：编码、字节序转换、数学工具（大整数、模逆、有限域运算等）

## 4. “执行过程可见及可测”的分级策略（选择 B：覆盖面 + 工作量平衡）

> 核心策略：**所有算法都做到 Level 1（可见 + 可测）**；选取代表性核心算法做到 **Level 2（关键中间步骤可视化 trace）**，用于满足“执行过程可见”的强证明，并在报告中重点展示。

### 4.1 Level 1：所有算法必须达到（全覆盖）
对每个算法/接口，至少提供以下“可见、可测”信息：
- 输入与输出可见：支持 `hex` / `base64` 形式展示（可配置）
- 参数可见：如模式、padding、迭代次数、输出长度等
- 状态码可见：`status_code` + `status_message`
- 可测：提供测试向量或固定样例用例，返回 `pass/fail` 与对比信息
- 运行统计：耗时 `time_ms`，便于对比与复现实验

### 4.2 Level 2：重点算法提供“关键过程 trace”
仅对以下算法提供更细粒度的“执行过程可视化”（trace=true 时返回），用于报告中重点展示：
- **AES**：KeyExpansion 轮密钥摘要；每轮 state（或按采样展示若干轮）；最终密文
- **SHA256**：padding 后关键字段；消息调度 W[t] 摘要；若干轮寄存器 a..h 的变化（可采样）；最终 digest
- **RSA / ECDSA（含 ECC-160bit）**：
  - RSA：密钥参数位数说明；签名/验签流程关键步骤摘要（padding/哈希标识/模幂运算信息）
  - ECC/ECDSA：曲线/位数说明；点加/倍点关键中间量摘要；签名 (r,s) 与验签判断过程摘要

> 注：trace 的目标是“可解释、可复现、可截图”，不追求把所有内部步骤全量输出；输出量过大时采用“采样 + 摘要 + 可配置轮数”的方式控制。

## 5. 功能与算法范围（覆盖作业技术范围）

### 5.1 对称加密算法（从零实现）
- AES（建议实现 AES-128；可扩展 192/256）
- SM4
- RC6

可见要点（Level 1）：明文/密钥/IV(如使用)、模式、padding、密文、耗时  
可视化（Level 2）：AES 轮函数关键过程（state/round key 摘要）

### 5.2 哈希 / HMAC / 派生（从零实现）
- SHA1、SHA256、SHA3、RIPEMD160
- HmacSHA1、HmacSHA256
- PBKDF2（基于 HMAC）

可见要点（Level 1）：输入、输出摘要、参数（PBKDF2: salt/iter/dkLen）、耗时  
可视化（Level 2）：SHA256 的 padding/消息调度/轮寄存器（采样）

### 5.3 编码算法
- Base64：建议自行实现编码/解码，并提供互逆测试
- UTF-8：展示字符串与字节序列（hex），并提供还原验证

### 5.4 公钥密码与签名（从零实现）
- RSA-1024bit（密钥生成、加/解密或签名/验签按要求实现）
- ECC-160bit（曲线与位数选择需在报告中说明）
- RSA-SHA1（签名/验签）
- ECDSA（签名/验签）

可见要点（Level 1）：key 参数长度、签名值、验签结果、耗时  
可视化（Level 2）：RSA/ECDSA 关键步骤摘要 trace

## 6. 接口化设计（满足“第三方程序调用 + 状态码返回”）

### 6.1 统一返回结构（JSON）
- `status_code`：0 表示成功；非 0 表示失败
- `status_message`：对错误原因的可读说明
- `result`：算法输出（密文/明文/摘要/签名/派生密钥/编码结果等）
- `trace`：可选，仅当 `trace=true` 时返回（Level 1 摘要或 Level 2 过程信息）
- `time_ms`：可选，运行耗时

### 6.2 统一输入约定（建议）
- 二进制相关输入统一采用 `hex` 或 `base64`（通过字段 `encoding` 指定）
- 文本输入允许 `text`，并明确字符集为 UTF-8

### 6.3 典型接口清单（示例，最终可在实现时细化）
- `/api/symmetric/encrypt`、`/api/symmetric/decrypt`
- `/api/hash/digest`
- `/api/hmac/compute`
- `/api/kdf/pbkdf2`
- `/api/encoding/base64/encode`、`/api/encoding/base64/decode`
- `/api/encoding/utf8/encode`、`/api/encoding/utf8/decode`
- `/api/asymmetric/rsa/keygen`、`/api/asymmetric/rsa/sign`、`/api/asymmetric/rsa/verify`
- `/api/asymmetric/ecc/keygen`、`/api/asymmetric/ecdsa/sign`、`/api/asymmetric/ecdsa/verify`
- `/api/selftest`：运行内置测试向量，输出每个算法 PASS/FAIL（“可测”的集中证明）
