# 状态码设计（Status Code Spec）：信息安全密码算法接口化实现与可测验证平台

> 目标：让第三方程序调用时能稳定判断执行结果与失败原因；同时便于报告展示“返回结果及执行状态代码”。

---

## 1. 设计原则

1. **0 表示成功**，其余均为失败或非预期状态。
2. **分段编码**：按错误类别划分区间，便于快速定位。
3. **稳定性优先**：状态码一旦定义尽量不改；错误信息放在 `status_message` 中。
4. **可测性友好**：SelfTest 中每个 case 的失败应能映射到明确状态码。

---

## 2. 状态码分段

- `0`：成功（OK）
- `1xxx`：请求与参数错误（客户端问题）
- `2xxx`：算法执行失败 / 校验失败（业务逻辑或数学条件不满足）
- `3xxx`：编码/解码与格式处理错误
- `9xxx`：系统内部错误（服务端异常）

---

## 3. 状态码明细表（建议初版）

### 3.1 成功
- `0`：OK

### 3.2 参数错误（1000–1999）
- `1001`：Invalid JSON（JSON 解析失败）
- `1002`：Missing required field（缺少必要字段）
- `1003`：Invalid field type（字段类型错误，如 iterations 不是整数）
- `1004`：Invalid value / out of range（数值越界，如 iterations<=0）
- `1005`：Unsupported algorithm（算法名不支持）
- `1006`：Unsupported mode/padding（模式/填充不支持）
- `1007`：Invalid key length（密钥长度不合法）
- `1008`：Invalid iv/nonce length（IV/Nonce 长度不合法）
- `1009`：Invalid encoding（encoding 不支持：非 hex/base64/text）
- `1010`：Invalid output encoding（output 不支持或与算法不兼容）
- `1011`：Trace level not supported（trace_level=2 但算法不支持；可选：降级返回 1 并用 message 提示）

### 3.3 算法执行/校验错误（2000–2999）
- `2001`：Decrypt failed（解密失败，如 padding 校验失败）
- `2002`：Verify failed（验签失败 / MAC 校验失败）
- `2003`：Invalid signature format（签名格式不正确）
- `2004`：RSA key invalid（RSA key 参数非法，如 n/e 不满足条件）
- `2005`：RSA keygen failed（RSA 密钥生成失败，重试次数耗尽）
- `2006`：ECC invalid curve params（曲线参数非法或不支持）
- `2007`：ECC point not on curve（点不在曲线上）
- `2008`：ECC scalar out of range（私钥 d 超范围）
- `2009`：Mod inverse not exist（模逆不存在，如 gcd(a,m)!=1）
- `2010`：PBKDF2 iterations too large（迭代次数过大导致策略限制触发，可选）

### 3.4 编码/解码错误（3000–3999）
- `3001`：Hex decode error（hex 非法字符/长度不对）
- `3002`：Base64 decode error（Base64 格式非法）
- `3003`：UTF-8 decode error（字节序列无法解码为 UTF-8）
- `3004`：Text encode error（文本编码异常，较少出现）

### 3.5 系统内部错误（9000–9999）
- `9001`：Internal error（未捕获异常）
- `9002`：Not implemented（接口存在但算法/功能尚未实现）
- `9003`：Timeout（超时，可选）

---

## 4. 响应字段约定

所有接口响应包含：
- `status_code`：整数
- `status_message`：字符串，面向人类可读、可截图写报告
- `result`：成功时为对象；失败时可为 `null` 或包含部分信息（如 `valid:false`）
- `trace`：当 `trace=true` 时返回（失败时也可返回已执行到的步骤，便于调试与“过程可见”）
- `time_ms`：耗时（即使失败也建议返回）

---

## 5. 状态码与 HTTP Status 的关系（建议）
为了便于第三方程序处理，可以采用：
- HTTP 永远返回 200，业务状态由 `status_code` 决定（简单一致）
或
- 参数错误返回 HTTP 400，算法校验失败返回 HTTP 200（valid=false），系统错误返回 HTTP 500（更 REST）

**建议初版**：HTTP 200 为主，减少前端/脚本处理复杂度；报告里强调“作业要求的执行状态代码为 status_code”。

---

## 6. 示例

### 6.1 成功
```json
{
  "status_code": 0,
  "status_message": "OK",
  "result": { "digest": "..." },
  "trace": null,
  "time_ms": 0.88
}
```

### 6.2 参数错误（缺字段）
```json
{
  "status_code": 1002,
  "status_message": "Missing required field: key",
  "result": null,
  "trace": null,
  "time_ms": 0.05
}
```

### 6.3 校验失败（验签不通过）
```json
{
  "status_code": 2002,
  "status_message": "Signature verification failed",
  "result": { "valid": false },
  "trace": { "steps": [ { "name": "digest_compare", "match": false } ] },
  "time_ms": 2.12
}
```
